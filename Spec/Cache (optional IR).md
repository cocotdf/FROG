<h2 id="overview">1. Overview</h2>

<p>
The <strong>cache</strong> mechanism provides optional, non-authoritative data used to speed up tooling workflows.
</p>

<p>
A typical use case is caching an intermediate representation (IR) generated from the source <code>.frog</code> file.
</p>

<p>
Cache data MUST NOT change program meaning.
Runtimes MUST treat cache data as optional and non-authoritative.
</p>

<hr/>

<h2 id="goals">2. Design goals</h2>

<ul>
  <li>Speed up loading, validation, and compilation workflows</li>
  <li>Allow tools to avoid recomputing IR when unchanged</li>
  <li>Remain safe to delete and regenerate</li>
  <li>Prevent cache data from redefining execution semantics</li>
  <li>Support stable version control workflows (cache can be excluded)</li>
</ul>

<hr/>

<h2 id="file-types">3. Cache storage options</h2>

<p>
FROG supports two cache storage options:
</p>

<ul>
  <li><strong>Preferred:</strong> a separate cache file (<code>.frog.cache</code>)</li>
  <li><strong>Optional:</strong> an embedded <code>cache</code> object inside <code>.frog</code></li>
</ul>

<p>
The separate cache file is preferred to avoid noisy diffs and merge conflicts in version control.
</p>

<hr/>

<h2 id="cache-file">4. <code>.frog.cache</code> file</h2>

<p>
A cache file is associated with a source file by name.
</p>

<pre>
example.frog
example.frog.cache
</pre>

<p>
The cache file MAY be generated by tooling and MAY be deleted at any time.
Tools SHOULD regenerate it automatically as needed.
</p>

<hr/>

<h2 id="embedded-cache">5. Embedded cache object (optional)</h2>

<p>
In some workflows, tools MAY embed cache data directly inside the <code>.frog</code> file.
</p>

<p>
If embedded, the cache object MUST be optional and non-authoritative.
Runtimes MUST ignore the embedded cache for correctness decisions.
</p>

<hr/>

<h2 id="structure">6. Cache object structure</h2>

<p>
Recommended structure:
</p>

<pre>"cache": {
  "cache_version": "0.1",
  "ir_version": "0.1",
  "generated_by": "frog-ide",
  "generated_at": "2026-03-05T12:00:00Z",
  "source_hash": "sha256:...",
  "ir": { }
}</pre>

<h3>Field definitions</h3>

<ul>
  <li><code>cache_version</code> — cache schema version (string)</li>
  <li><code>ir_version</code> — IR schema version (string)</li>
  <li><code>generated_by</code> — tool identifier (string)</li>
  <li><code>generated_at</code> — ISO 8601 timestamp (string)</li>
  <li><code>source_hash</code> — hash of canonical source content (string)</li>
  <li><code>ir</code> — cached IR payload (object)</li>
</ul>

<p>
The <code>source_hash</code> allows tools to detect stale caches reliably.
</p>

<hr/>

<h2 id="authoritativeness">7. Non-authoritative rules</h2>

<p>
Cache data MUST be treated as an optimization only.
</p>

<ul>
  <li>Runtimes MUST NOT require cache data to execute a program.</li>
  <li>Runtimes MUST NOT trust cache data for correctness.</li>
  <li>Tools MUST be able to rebuild cache data from the source <code>.frog</code>.</li>
</ul>

<p>
If cache data conflicts with source content, the source content MUST win.
</p>

<hr/>

<h2 id="invalid-cache">8. Invalid or stale cache handling</h2>

<p>
Cache data is considered stale if:
</p>

<ul>
  <li><code>source_hash</code> does not match the current source content</li>
  <li><code>ir_version</code> is not supported</li>
  <li>the IR payload fails validation</li>
</ul>

<p>
When stale:
</p>

<ul>
  <li>Tools SHOULD discard the cache and regenerate it.</li>
  <li>Runtimes MUST ignore the cache and operate from source-derived IR.</li>
</ul>

<hr/>

<h2 id="security">9. Security considerations</h2>

<p>
Cache content MUST NOT be trusted as executable truth.
</p>

<ul>
  <li>Tools SHOULD validate cache payloads before use.</li>
  <li>Runtimes MUST validate IR regardless of cache presence.</li>
  <li>Tools SHOULD avoid storing sensitive information in cache.</li>
</ul>

<p>
Cache data MAY be considered untrusted input.
</p>

<hr/>

<h2 id="validation">10. Validation rules</h2>

Implementations MUST enforce:

<ul>
  <li>If present, <code>cache</code> MUST be a JSON object.</li>
  <li>If present, <code>cache.ir</code> MUST be a JSON object.</li>
  <li>If present, <code>cache.ir_version</code> MUST be a string.</li>
  <li>If present, <code>cache.source_hash</code> SHOULD be present to detect staleness.</li>
  <li>Runtimes MUST ignore cache data for correctness decisions.</li>
</ul>

  }
}</pre>

<hr/>

<h2 id="summary">12. Summary</h2>

<p>
Cache data provides optional acceleration for tooling workflows.
</p>

<ul>
  <li>Preferred storage is a separate <code>.frog.cache</code> file.</li>
  <li>Embedded cache is allowed but discouraged for version control stability.</li>
  <li>Cache data is non-authoritative and safe to ignore.</li>
  <li>Source content always defines program meaning.</li>
</ul>
